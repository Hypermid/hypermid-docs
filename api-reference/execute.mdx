---
title: "Execute Swap"
openapi: "POST /v1/execute"
description: "Execute a cross-chain swap"
---

Initiates a cross-chain swap execution. Returns a `transactionRequest` to be signed and sent by the user's wallet, or a `depositAddress` for manual deposit flows.

For SuperSwap routes, the response contains the bridge transaction for the user to sign. The backend automatically handles cross-chain delivery and destination swaps.

<ParamField body="fromChain" type="number" required>
  Source chain ID.
</ParamField>

<ParamField body="toChain" type="number" required>
  Destination chain ID.
</ParamField>

<ParamField body="fromToken" type="string" required>
  Source token contract address.
</ParamField>

<ParamField body="toToken" type="string" required>
  Destination token contract address.
</ParamField>

<ParamField body="fromAmount" type="string" required>
  Amount to swap in the smallest unit.
</ParamField>

<ParamField body="fromAddress" type="string" required>
  Sender wallet address on the source chain.
</ParamField>

<ParamField body="toAddress" type="string" required>
  Recipient wallet address on the destination chain.
</ParamField>

<ParamField body="slippage" type="number">
  Maximum slippage tolerance. Default: `0.03`.
</ParamField>

<ParamField body="tool" type="string">
  Specific tool to use. If omitted, the best tool is selected automatically.
</ParamField>

<ParamField body="depositMode" type="string">
  Override deposit mode. Set to `"manual"` to force manual deposit flow for intent-based routes.
</ParamField>

## Multi-step Swap Execution

For multi-step swaps (where the quote has `singleSignature: true` and `steps[1].automated: true`), the execution flow requires an additional confirmation step:

1. **Sign and broadcast** the transaction from `steps[0].transactionRequest`
2. **Wait for confirmation** on the source chain
3. **Sign an EIP-712 authorization message** (gasless wallet signature — no transaction cost)
4. **Call `POST /v1/inbound-receiver/register`** with the tx hash + signature

The backend then automatically handles bridging and destination-chain execution.

### EIP-712 Authorization

The quote response includes the full EIP-712 signing specification in `afterStep1.eip712`. The wallet must sign a typed data message authorizing the routing parameters:

```typescript
// After Step 1 confirms:
const { afterStep1 } = quote;

const signature = await walletClient.signTypedData({
  domain: afterStep1.eip712.domain,
  types: afterStep1.eip712.types,
  primaryType: afterStep1.eip712.primaryType,
  message: {
    ...afterStep1.eip712.message,
    txHash: step1TxHash,  // Actual tx hash from Step 1
  },
});

// Confirm the deposit
await fetch("https://api.hypermid.io/v1/inbound-receiver/register", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    ...afterStep1.body,
    txHash: step1TxHash,
    signature,
  }),
});
```

The confirm endpoint verifies the transaction on-chain and validates the wallet signature before starting the automatic bridge.

<CodeGroup>
```typescript TypeScript
const exec = await client.execute({
  fromChain: 1,
  toChain: 42161,
  fromToken: "0x0000000000000000000000000000000000000000",
  toToken: "0xaf88d065e77c8cC2239327C5EDb3A432268e5831",
  fromAmount: "1000000000000000000",
  fromAddress: "0xYourAddress",
  toAddress: "0xYourAddress",
  slippage: 0.03,
});

if (exec.data.transactionRequest) {
  // Sign and send the transaction
  const tx = await wallet.sendTransaction(exec.data.transactionRequest);
  console.log("Tx hash:", tx.hash);
} else if (exec.data.depositAddress) {
  // Manual deposit: show deposit info to user
  console.log("Send to:", exec.data.depositAddress);
  console.log("Memo:", exec.data.memo);
}
```

  The `transactionRequest` data is time-sensitive. Execute the transaction within a few minutes of receiving the response, as prices and gas estimates may change.
</Warning>
